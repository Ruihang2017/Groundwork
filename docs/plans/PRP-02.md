### 2.1 `lib/rehearse/prompt.ts` (Deliverable 1) — new

- `export const REHEARSE_SYSTEM_PROMPT: string` — a template literal. Required content (each clause is pinned by a prompt-invariant test in §3):
  1. **Task**: generate an interview-rehearsal brief for ONE candidate about to interview for ONE role, given a structured JD reading, the CROSS ledger (bindings + gaps), the candidate's real project library, and optional company intel. The point (PRD §1 F3): interviews are decided **on the gaps**, so anchor rehearsal on the ledger's gaps and the candidate's real project detail — not on a generic question bank.
  2. **Output contract**: ONE JSON object, no prose, no code fence, with a worked example showing all keys: `questions` (an array of exactly 5 `{ projectId, question, trap }`), `askThem` (exactly 3 strings), `positioning` (a string). State the counts literally: **exactly 5 questions, exactly 3 askThem** (PRD §5.4).
  3. **`questions` rules**: exactly 5. Each `projectId` **must be an `id` copied verbatim from `<library>`** — a projectId not in the library is discarded by the server before the user sees it, so cite only real ones (this is what keeps the referential-integrity drop count at 0 on good output). Each `question` must be **specific to this candidate's actual project** — anchored in a concrete detail from that project's `summary`/`stack`/`metrics`, ideally targeting one of the ledger's **gaps** — such that it **could not be meaningfully asked of a random candidate with the same title** (the Q3 specificity criterion, PRD §6/§13). Each `trap` is the **second question an interviewer asks after the textbook answer** — the follow-up that separates real experience from a rehearsed line; non-empty and specific.
  4. **`askThem` rules**: exactly 3 questions the CANDIDATE should ask the interviewer that **they could only formulate by having done research** — grounded in `<intel>` (company snapshot / recent items / engineering signals) when present. If `<intel>` says intel is unavailable, ground them instead in specifics of `<jd_extract>`/`<ledger>` — still never generic ("what's the culture like", "what's a typical day"). The test: someone who did no preparation could not ask it.
  5. **`positioning`**: a short paragraph on how this candidate should position themselves for THIS role — leaning on their strongest bindings and naming how to bridge the top gaps. Non-empty.
  6. **Retrieve, don't generate** (PRD §2 P1/P2): every question and talking point must trace to `<library>`/`<ledger>`/`<jd_extract>`/`<intel>`. Never invent a project detail, a company fact, or a metric the inputs do not contain.
  7. **Language** (D9): write every output string in the **same language as the requirement texts inside `<jd_extract>`**; do not translate or mix.

## 4. Risks and edge cases

**Security-sensitive paths (the Reviewer should start here):**

- **S1 — prompt injection via `intel` (PRD §12).** `intel` is the one input here that transited the open internet (RESEARCH's `web_search`), and it arrives in the **request body** — so it is doubly untrusted (web-sourced AND client-supplied; a direct API client can POST any `Intel`-shaped object, bypassing PRP-01 entirely). Mitigations: (a) the `<intel>` delimiter + the explicit untrusted-content clause naming it (§2.1 clause 8); (b) the output is Zod-parsed into a fixed shape and consumed only as data (returned, written to `jsonb`, never executed/interpolated into SQL); (c) the NUL guard on `body.intel` (D8) before it can reach the `jsonb` write. **Residual, accepted:** a poisoned `intel` can still steer `askThem` content — the same "面前人工过一遍" product norm PRD §12 states applies.
- **S2 — cross-user isolation (PRD §8.3).** `userId` comes only from `requireUserId()`. Every DB read is scoped: `getJob(userId, id)` (404 covers foreign), `getLibrary(userId)`, `getBrief` joins through `jobs.userId`. `upsertBrief` takes **no `userId`** and does **no** ownership check — so **step 3's `getJob` is the sole ownership gate for the write** (comment it at the `upsertBrief` call). A test seeds another user's job and confirms the 404.
- **S3 — logging discipline.** The 422 and repair paths are the chattiest; they must log **reasons/counts/paths only** — never the reply text, the `Rehearse`/`Brief`/`intel`, `job.jd`/`jdRaw`/`ledger`, the library, `company`/`role`, or headers (they carry `ANTHROPIC_API_KEY`). A raw pg/Drizzle error is never logged whole (`briefs.ts` `parseRow` logs Zod paths only).
- **S4 — caching.** `Cache-Control: no-store` on the 200 — the body carries the user's brief; a shared cache holding it is a cross-user leak.
- **S5 — CSRF.** Auth.js v5 cookie defaults (`httpOnly`, `sameSite: 'lax'`) ⇒ a cross-site POST carries no session cookie and gets 401 before any spend. No extra token.

**Correctness / cost / operability:**

- **R1 — the free-replay paid-call vector (THE sharpest risk; D13).** `prep` is charged upstream at RESEARCH, and this route charges nothing, so an unguarded, overwriting REHEARSE can be re-POSTed indefinitely per single `prep` unit — unbounded paid (token-cost) calls, bounded only by the org-wide breaker. This is structurally the vector FIT-02 closed with `409 already_fitted`; the ticket's explicit overwrite requirement is why this plan does **not** close it here. Mitigations in place: the org-wide breaker (step 7), the single-flight instruction to PRP-03/PRP-04 (route header), and the fact that the intended client flow always pairs REHEARSE with a fresh (quota-charged) RESEARCH. **Escalated to Horace (§5 Q2); do not add a guard silently — it contradicts the ticket.**
- **R2 — the D5 schema tension is load-bearing and cross-cutting.** A persisted Brief with < 5 questions is not a valid FND-03 `Rehearse`. This plan contains it module-locally, but three consumers are affected and the Reviewer must check them: (a) `assertQ1Questions` **fails** on < 5 (so a real REHEARSE that drops a question fails Q1 — the fixture corpus must pair libraries whose ids the model will actually cite, or Q1 flakes for a non-quality reason); (b) PRP-04 must read the Brief via `getBrief` (relaxed) and must **not** re-parse it against the strict FND-03 `Brief`; (c) any future consumer importing `Brief` directly. **The real fix is an FND-03/Horace schema decision — §5 Q1 / §6 ADR-A.**
- **R3 — the wasted `prep` unit (ticket Feedback obligation #3).** A REHEARSE that 422s after a successful RESEARCH consumes the day's `prep` unit with no Brief. **Escalate via `06-prep/README.md` open question #3; do not add a refund/retry here.**
- **R4 — `briefs.jobId` has no UNIQUE constraint** (verified — only `briefs_job_id_idx`). Two simultaneous REHEARSE for one job can both read "no row" and both INSERT (two rows). Accepted for v1 (single-user single-session assumption, identical to TLR-01's `tailored_resumes` caveat); `getBrief`'s `ORDER BY updatedAt DESC LIMIT 1` makes reads deterministic (newest wins). The real fix (UNIQUE + migration) is **FND-05 file-scope** — do not add it here.
- **R5 — a 422 is UNMETERED** (D11, carried from FIT-02/PRP-01). No `usage_events` row ⇒ the breaker under-counts REHEARSE's (token-only) spend. Repo-wide rule: all stage routes change together (§5 Q3); do not start recording `status:'failure'` here.
- **R6 — in-brief language mix (D9).** A non-English JD yields JD-language `questions`/`positioning` but the `intel` (hence some `askThem` phrasing) is English (PRP-01 D9d). Accepted for v1; reversible via a language hint threaded through PRP-01→PRP-02→PRP-04 (§5 Q4).
- **R7 — `max_tokens` truncation on a large library.** A very large `<library>` could crowd the 4096 output budget and truncate the `Rehearse` (→ repair → possibly 422). Mitigation: the measured smoke run (§2.1) sets whether 4096 needs raising — a one-constant lever, not a redesign.
- **R8 — Next 15 async `params` / a reintroduced static `@/db/index` import.** Both type-check in isolation and fail only under `corepack pnpm build` / the build-guard test. Run the build before calling the ticket done (§7).
- **R9 — `assertQ1Questions`/`assertQ3Specific` are imported from `@/eval` (`.ts`-extension re-exports).** They resolve fine under Vitest's `@` alias; do **not** attempt to run these fixture tests under plain Node. Keep them inside the Vitest route test file (in write-scope), never in `eval/**`.
